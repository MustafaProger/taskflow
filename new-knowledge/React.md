### 1. Что такое `event` в React и как его понимать в обработчиках?

`event` в React - это объект **с данными о событии**, которые React передает в обработчик в момент срабатывания.

### Важно: событие vs event vs обработчик

Чтобы не путаться, важно четко различать три вещи:

- событие — это действие (клик, нажатие клавиши, завершение анимации)
- event — это объект с данными об этом событии
- обработчик (handler) — это функция, которая реагирует на событие

Пример:

```tsx
<button onClick={handleClick}>Нажми</button>
```

Здесь:

- событие — клик
- обработчик — `handleClick`
- event — объект, который React передаст в `handleClick`

Коротко:
событие происходит → React вызывает обработчик → передает в него `event`

Проще всего думать так:

- в JSX мы передаем не само событие, а функцию-обработчик
- когда событие реально происходит, React вызывает эту функцию
- в момент вызова React подставляет в нее `event`

Пример:

```tsx
<button onClick={handleClick}>Нажми</button>
```

```tsx
function handleClick(event: React.MouseEvent<HTMLButtonElement>) {
  console.log(event.type)
}
```

Что здесь происходит по шагам:

1. Мы передали функцию `handleClick` в `onClick`.
2. Пользователь нажал на кнопку.
3. React вызвал `handleClick(event)`.
4. Внутри `event` лежит информация о клике.

Важно: `event` не хранится в `onClick` заранее. Он создается и передается только тогда, когда событие действительно произошло.

### Что такое `event` на практике

Обычно в React это объект события с удобным общим интерфейсом для работы в обработчиках. Через него можно понять:

- что именно произошло
- на каком элементе это произошло
- нужно ли отменить стандартное поведение
- нужно ли остановить всплытие события

Часто используют такие свойства и методы:

```ts
event.target
event.currentTarget
event.type
event.preventDefault()
event.stopPropagation()
```

### Разница между `event.target` и `event.currentTarget`

- `event.target` — элемент, на котором событие произошло фактически
- `event.currentTarget` — элемент, на котором висит обработчик

Пример:

```tsx
<div onClick={handleClick}>
  <button>Нажми</button>
</div>
```

Если кликнуть на кнопку:

- `event.target` → `<button>`
- `event.currentTarget` → `<div>`

### Что делают `preventDefault` и `stopPropagation`

`event.preventDefault()` — отменяет стандартное поведение браузера

Примеры:

- форма не отправится
- ссылка не откроется

`event.stopPropagation()` — останавливает всплытие события вверх по DOM

Пример:
если клик был внутри кнопки, но есть обработчик у родителя, можно остановить всплытие и родитель не получит событие

### Что у событий общее, а что разное

У разных событий есть общая база, но дополнительные поля зависят от типа события.

У клика мышью могут быть полезны:

```ts
event.clientX
event.clientY
```

У клавиатурного события:

```ts
event.key
event.code
```

У завершения CSS-перехода:

```ts
event.propertyName
event.elapsedTime
```

Главная мысль:

- базовые свойства есть у многих событий
- специальные свойства зависят от конкретного типа события
- поэтому тип обработчика должен соответствовать событию

Поэтому важно:
тип `event` в TypeScript зависит от события

```ts
onClick → React.MouseEvent
onKeyDown → React.KeyboardEvent
onTransitionEnd → React.TransitionEvent
```

### Пример с `onTransitionEnd`

```tsx
<span onTransitionEnd={handleIndicatorTransitionEnd} />
```

```tsx
function handleIndicatorTransitionEnd(
  event: React.TransitionEvent<HTMLSpanElement>
) {
  console.log(event.propertyName)
}
```

Когда transition у `span` закончится, React вызовет обработчик и передаст в него `event`.

В этом примере:

- `event` говорит, что переход завершился
- `event.propertyName` показывает, какое CSS-свойство завершило анимацию

Если завершился переход свойства `transform`, то `event.propertyName` будет `"transform"`.

Если сама запись `React.TransitionEvent<HTMLSpanElement>` читается тяжело, это уже вопрос не про поведение React, а про типизацию обработчика в TypeScript.

Подробный разбор этой записи вынесен в [TypeScript.md](./TypeScript.md#typescript-q2).

### Если нужно передать и `event`, и свои данные

Иногда в обработчик нужно передать не только `event`, но и, например, `id`.
Тогда делают промежуточную функцию:

```tsx
<button onClick={(event) => handleClick(event, taskId)}>Нажми</button>
```

```tsx
function handleClick(
  event: React.MouseEvent<HTMLButtonElement>,
  id: string
) {
  console.log(id, event.type)
}
```

Важно понимать:
`event` всегда передается первым аргументом автоматически.
Мы просто оборачиваем вызов, чтобы передать дополнительные данные.

То есть `event` не нужно "доставать" вручную. React сам передаст его в первый аргумент обработчика.

### Короткая формулировка для запоминания

`event` - это объект с данными о событии, который React автоматически передает в обработчик в момент срабатывания.

Коротко:

- `onClick`, `onChange`, `onTransitionEnd` и похожие пропсы принимают обработчики событий
- React сам передает `event` в обработчик
- у событий есть общие свойства
- у каждого типа события есть свои дополнительные поля

### 2. Как устроены `props` и где тут `children`?

`props` - это весь объект данных, который приходит в компонент.

`children` - это одно из полей внутри `props`.

То есть правильно думать так:

```ts
props = {
  wrapClassName,
  surfaceClassName,
  children
}
```

А не так, будто `children` существует отдельно от `props`.

Пример:

```tsx
<GlassSurface wrapClassName="outer">
  <span>Hello</span>
</GlassSurface>
```

React примерно превращает это в такой объект:

```ts
{
  wrapClassName: "outer",
  children: <span>Hello</span>
}
```

Главная идея:
все, что написано между открывающим и закрывающим тегом компонента, попадает в `children`.

В этом проекте компонент `GlassSurface` получает `children` и выводит их внутри блока `.glass-surface`.

Важно помнить:

- `props` - это все входные данные компонента
- `children` - это только одна часть этих данных
- если между тегами ничего нет, то `children` будет пустым

Коротко:  
`children` не живет отдельно.  
Это обычное поле внутри `props`.

### 3. Какие бывают категории компонентов в React-проекте?

Здесь важно понимать главное:
нет одной единственно правильной схемы, по которой обязаны называться все компоненты.

Но на практике часто используют разделение по роли компонента.

Один из самых понятных вариантов такой:

- `ui`
- `layout`
- `feature`

### `ui`

Это простые визуальные компоненты.
Они обычно отвечают за внешний вид и не знают про бизнес-логику.

Примеры:

- `Button`
- `Input`
- `Card`
- `GlassSurface`

Идея простая:
`ui`-компонент показывает интерфейс и принимает данные через `props`.

### `layout`

Это компоненты-контейнеры, которые раскладывают части интерфейса по местам.
Они задают каркас страницы или секции.

Примеры:

- `Header`
- `Sidebar`
- `PageLayout`
- `Section`

Идея простая:
`layout` отвечает за структуру, расположение блоков и композицию.

### `feature`

Это компоненты, связанные с конкретным пользовательским действием или бизнес-задачей.
Внутри у них уже может быть логика, состояние, обработчики и работа с данными.

Примеры:

- `AuthButton`
- `SearchForm`
- `ProfileMenu`
- `TaskFilter`

Идея простая:
`feature` не просто показывает интерфейс, а решает конкретную задачу.

## Как на это смотреть в реальности

Полезное правило такое:

- если компонент в основном показывает внешний вид, это ближе к `ui`
- если компонент раскладывает другие части интерфейса, это ближе к `layout`
- если компонент реализует поведение или бизнес-сценарий, это ближе к `feature`

Пример:

```tsx
<Header>
  <Logo />
  <AuthButton />
</Header>
```

Здесь:

- `Header` обычно `layout`
- `Logo` обычно `ui`
- `AuthButton` может быть `feature`

Важно:
наличие логики рядом не делает весь родительский компонент `feature`.
Смотреть нужно на ответственность самого компонента.

## Какие еще подходы встречаются

Кроме `ui`, `layout` и `feature`, в проектах еще можно встретить:

- `pages`
- `widgets`
- `shared`
- `entities`

Это уже зависит от выбранной архитектуры проекта.

Коротко:
для начала достаточно уверенно различать три роли:
`ui`, `layout`, `feature`.

### 4. Как работает цикл рендера в React (render → commit → paint → effects)?

Это основной жизненный цикл обновления интерфейса в React.

Короткая формула:
Render → Commit → Paint → Effects

### 1. Render (вычисление)

React вызывает компонент как функцию и получает JSX.

Важно:

- на этом этапе DOM НЕ изменяется
- это просто вычисление "какой UI должен быть"

Пример:

```tsx
const App = () => <div>Hello</div>;
```

### Diff (сравнение внутри render)

React сравнивает:

- предыдущий результат
- новый результат

Примеры:

- первый рендер: было `null` → стало `<div>Hello</div>`
- обновление: `<div>0</div>` → `<div>1</div>`

React определяет, ЧТО именно изменилось.

Важно:
diff — это не отдельный этап, а часть подготовки к commit.

### 2. Commit (изменение DOM)

React применяет изменения в реальный DOM.

- при первом рендере — создаёт DOM с нуля
- при обновлении — меняет только разницу

Важно:
commit происходит ТОЛЬКО если есть изменения.

### 3. Paint (отрисовка браузером)

Браузер рисует обновлённый DOM на экране.

Важно:
если commit не было → paint не будет.

### 4. Effects

После commit React запускает побочные эффекты:

- useEffect

### Первый рендер

1. React вызывает компонент
2. получает JSX
3. сравнивает с `null`
4. создаёт DOM (commit)
5. браузер рисует (paint)
6. запускаются эффекты

### Обновление (после setState)

1. происходит событие
2. вызывается setState
3. React снова вызывает компонент (render)
4. сравнивает старый и новый результат (diff)
5. если есть изменения → commit
6. браузер рисует (paint)
7. запускаются эффекты

### Важные правила

- Render происходит всегда
- Commit — только если есть изменения
- Paint зависит от commit
- Effects запускаются после commit

### 5. Что такое `useLayoutEffect` и чем он отличается от `useEffect`?

`useLayoutEffect` - это React-хук, который срабатывает после того, как React уже обновил DOM, но до того, как браузер показал результат пользователю.

Его используют в тех случаях, когда нужно:

- измерить элемент
- узнать его ширину, высоту или позицию
- сразу подвинуть другой элемент без визуального скачка

Пример из проекта:
в `TaskState` активная кнопка сначала появляется в DOM, потом через `useLayoutEffect` можно узнать ее `offsetLeft` и `offsetWidth`, а потом поставить индикатор точно под эту кнопку.

### Как работает `useLayoutEffect`

Порядок такой:

1. React делает рендер
2. DOM обновляется
3. срабатывает `useLayoutEffect`
4. если внутри поменялся `state`, React сразу делает еще один рендер
5. только потом браузер показывает итоговый результат

Из-за этого пользователь не видит промежуточное неправильное положение элемента.

### Чем отличается от `useEffect`

`useEffect` тоже срабатывает после рендера, но позже:
он запускается уже после того, как браузер показал интерфейс.

Из-за этого разница такая:

- `useLayoutEffect` подходит для измерения DOM и точной визуальной настройки
- `useEffect` подходит для запросов, подписок, таймеров и другой побочной логики

Если измерять DOM через `useEffect`, интерфейс иногда может сначала показаться в одном виде, а потом резко перестроиться.
Это выглядит как мигание или скачок.

### Главное различие

- `useLayoutEffect` нужен, когда важно успеть измерить или поправить интерфейс до показа в браузере
- `useEffect` нужен, когда код можно выполнить уже после показа интерфейса

Коротко:
если работа связана с размерами, позицией и DOM-измерениями, чаще нужен `useLayoutEffect`.

### 6. Что такое хуки (hooks) в React и как их понимать?

> Хук (hook) — это функция, через которую компонент подключается к возможностям React: состоянию, жизненному циклу, контексту и другим механизмам.

Главная идея:
хук — это не просто функция, а интерфейс доступа к внутренней системе React.

### Почему без хуков нельзя управлять состоянием

Если написать обычную переменную:

```ts
let count = 0;
```

React не будет отслеживать её изменения:

- не сохранит значение между рендерами
- не вызовет повторный рендер

С хуком:

```ts
const [count, setCount] = useState(0);
```

Теперь:

- React хранит значение
- React вызывает повторный рендер при изменении
- состояние становится частью системы React

### Почему хук называется "hook"

Слово "hook" переводится как "крючок".

Идея в том, что компонент "зацепляется" за систему React:

```
Компонент → хук → система React
```

### Важные свойства хуков

1. Это функции (начинаются с `use`)
2. Вызываются только внутри компонентов или других хуков
3. Дают доступ к возможностям React
4. Связаны с порядком вызова (поэтому нельзя вызывать их в условиях или циклах)

### Примеры хуков

```ts
useState      // состояние
useEffect     // побочные эффекты
useLayoutEffect // работа с DOM до показа
useContext    // доступ к контексту
```

### Кастомные хуки

Можно создавать свои хуки:

```ts
function useSomething() {
  // внутренняя логика
}
```

Такие хуки позволяют переиспользовать логику.

### Связь с Zustand

Zustand использует ту же концепцию.

Когда ты пишешь:

```ts
const tasks = useTaskStore(state => state.tasks);
```

`useTaskStore` — это тоже хук.

Он:

- подписывает компонент на глобальное состояние
- вызывает ререндер при изменениях
- работает по тем же принципам, что и React-хуки

### Разница между хуком и обычной функцией

Хук:

```ts
const value = useTaskStore(...)
```

- подключает к системе состояния
- влияет на рендер

Обычная функция:

```ts
function getTasks() {}
```

- просто выполняет код
- не связана с React
- не вызывает ререндер

### Короткая формулировка для запоминания

Хук — это функция, через которую компонент подключается к возможностям React (состояние, жизненный цикл и т.д.).
